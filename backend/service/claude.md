# backend/service

## Назначение

Слой данных и бизнес-логики. Управляет PostgreSQL-соединениями, кешированием и основными бизнес-процессами.

## Файлы

### db.py — Подключение к БД и схема
- Управление пулом asyncpg (10–20 соединений)
- Система миграций (таблица `versions`)
- **Схема БД (v1.0.1):**

| Таблица | Назначение |
|---|---|
| `users` | Профили (Telegram ID, имя, баланс, реферер) |
| `user_onboardings` | Сессии онбординга с анкетами (JSONB) |
| `user_tasks` | Задачи поиска лидов с тегами (JSONB) |
| `user_channels` | Telegram-каналы для доставки рекомендаций |
| `recommendations` | Лиды с метаданными (JSONB) |
| `user_subscriptions` | Подписки (trial/active, даты) |
| `transactions` | Платежи (статус, шлюз, суммы) |
| `task_statistics` | Ежедневная статистика лидов по задачам |
| `user_task_rules` | Правила фильтрации (text[]) |

- **Кеши**: users (300с), onboarding (20мин), rules (1ч)

### users.py — CRUD пользователей
- `create_user()` — регистрация из Telegram
- `fetch_user_by_id()` — с LRU-кешем
- `patch_user_by_id()` — обновление профиля

### finder.py — Задачи и рекомендации
- CRUD для задач (лимит 5 на пользователя)
- Управление каналами (привязка/отвязка)
- Управление онбордингом (создание, ответы, завершение, удаление)

### subscriptions.py — Подписки и платежи
- Управление подписками (trial, активация, продление)
- Инициация покупки через Lava/Alpha
- Обработка вебхуков платёжных шлюзов
- Фоновая задача `check_transactions()` — каждые 10 мин проверяет зависшие транзакции
- Логика баланса: если сумма не совпадает с тарифом → зачисление на баланс

### refs.py — Рефералы
- `fetch_user_refs()` — список рефералов с информацией о биллинге

### task_rules.py — Правила фильтрации
- `save_user_rules()` — сохранение извлечённых правил
- `fetch_user_rules()` — получение с кешированием (1 час)
